# 包租代管與代租代管系統規劃

版本：第一版規劃草案

目的：建立一套可管理「包租代管」與「代租代管」的內部管理系統，並能依每個房子、房間、案件計算收支、前置投入、家電出資、回本月份與投報率。

## 1. 系統定位

第一版先做 Web 管理後台，手機也能使用，但不先做原生手機 App。

建議技術方向：

- 前端與後端：Next.js
- 資料庫：PostgreSQL
- 登入與權限：Supabase Auth 或自建登入系統
- 檔案儲存：Supabase Storage、S3 或 Cloudflare R2
- 部署：Vercel、Railway、Render 或 VPS

## 2. 第一版功能

- 房東管理
- 房客管理
- 物件管理
- 房間或戶別管理
- 包租代管案件
- 代租代管案件
- 租約管理

- 房東撥款
- 支出與代墊
- 修繕工單

第一版暫不優先做：線上刷卡、電子簽約、原生手機 App、AI 自動估租、政府系統串接、完整會計總帳。

## 3. 使用者角色

- 管理員：管理所有資料、權限、系統設定與報表。
- 業務或物件管理人員：建立房東、房客、物件、案件、租約、修繕紀錄、格局圖與照片。
- 會計：管理收租、房東撥款、支出、代墊與匯出報表。
- 房東：第二階段開放，可查看出租狀態、撥款、修繕紀錄、格局圖與報表。
- 房客：第二階段開放，可查看租約、上傳付款證明、報修與查看繳費紀錄。

## 4. 兩種案件模式

系統核心要有 `management_type` 欄位，用來區分案件模式。

```text
master_lease = 包租代管
agency_management = 代租代管
```

包租代管：公司向房東承租，再轉租給房客。公司通常承擔空租風險，收入主要來自租差，也要承擔家電、裝修、修繕、空租等成本。

代租代管：公司協助房東出租與管理。租金收入主要屬於房東，公司收取管理費、媒合費、服務費，修繕可能由公司代墊後再向房東請款。

## 5. 平面圖與類 3D 格局圖

此功能建議第一版就加入，原因是它會影響出租效率、修繕定位、家具家電配置與前置投入評估。

可上傳的檔案：

- 2D 平面圖圖片：JPG、PNG、WEBP
- PDF 格局圖
- 類 3D 格局圖圖片
- 裝修前後格局圖
- 家具配置圖
- 水電管線圖，第二階段可加
- 互動式 3D 模型，第二階段可加

第一版功能：

- 每個物件可上傳多張格局圖。
- 每個房間或戶別可指定主要格局圖。
- 可標記格局圖類型，例如 2D、類 3D、家具配置、裝修前、裝修後。
- 可設定是否公開給房東或房客看。
- 可與修繕工單連結，用來標示問題位置。
- 可與家電設備連結，用來記錄設備放置區域。

第二階段可擴充：

- 在格局圖上標記冷氣、冰箱、洗衣機、熱水器、床、衣櫃等設備位置。
- 在格局圖上標記報修位置。
- 依格局圖建立房間面積與出租單位。
- 3D 看屋或類 3D 視覺化展示。
- 自動產生出租頁面用的圖文資料。

## 6. 系統模組

- 基礎資料：房東、房客、員工、物件、房間或戶別。
- 案件管理：包租代管案件、代租代管案件、負責人、案件期間、案件狀態。
- 租約管理：房東合約、房客租約、租金、押金、付款週期、合約附件、到期提醒。
- 收支管理：應收租金、實收租金、未收款、房東撥款、管理費、媒合費、支出、代墊。
- 家電與前置投入：家電、家具、裝修、清潔、仲介費、拍照費、出資方、使用年限、每月攤提。
- 格局圖管理：平面圖、類 3D 圖、家具配置圖、裝修前後圖、公開權限、關聯房間。
- 修繕管理：報修、處理狀態、維修廠商、預估金額、實際金額、付款方、是否代墊、照片附件。
- 報表：每房損益、每案損益、租差報表、服務收入報表、回本月份、年化投報率、空租天數、應收未收、房東撥款。

## 8. 投報率計算

包租代管：

```text
每月淨利 =
房客租金收入
- 給房東租金
- 修繕費
- 管理成本
- 空租損失
- 家電折舊
- 前置投入攤提

年化投報率 =
每月淨利 * 12 / 公司投入成本 * 100%
```

代租代管：

```text
每月淨利 =
管理費收入
+ 媒合費攤提
+ 其他服務費
- 人力成本
- 行政成本
- 未回收代墊
- 修繕處理成本

年化投報率 =
每月淨利 * 12 / 公司投入成本 * 100%
```

## 9. 每月結算設計

系統不建議每次打開報表都重新計算所有歷史資料。建議每個月產生一次 `property_roi_snapshots`，固定保存該月收入、支出、家電折舊、前置投入攤提、淨利、累積獲利、回本月份與年化投報率。

## 11. 開發順序

1. 確認規劃與資料庫。
2. 建立內部管理後台。
3. 建立物件、房間、案件、租約。
4. 建立格局圖與照片上傳。
5. 建立收支、修繕、家電與前置投入。
6. 建立每月結算與投報率報表。
7. 第二階段再開放房東入口與房客入口。

## 12. 可變更原則

這份規劃是第一版藍圖，不是不能改的規格。後續可以調整欄位名稱、資料表拆分、投報率公式、案件流程、權限、報表格式、是否加入手機 App、是否串接金流或電子簽約。

建議保留的核心：

- 每個案件都要有案件模式。
- 每個收入與支出都要能追到案件。
- 家電與前置投入要能分攤。
- 格局圖要能追到物件、房間或案件。
- 投報率要用每月快照保存。
- 包租代管與代租代管的收入邏輯要分開。
